# NPU 统一推理算力池需求说明(v2)

> 本文是对《NPU 统一卡池多业务调度需求说明》的重写。经过对实际负载的澄清,问题性质发生了根本变化:它**不是一个 HPC 式的"两类业务抢卡"调度问题,而是一个"统一多模型推理算力池"的编排问题**。原文中大量针对训练型长作业的机制(checkpoint 状态机、被抢占恢复、保底/弹性/最大三层百分比配额、P4>P3>P2 抢占顺序)在本场景下基本不成立,已在本版中移除,并在第 10 章说明原因。

---

## 1. 背景与重新定位

### 1.1 真实负载

统一 NPU 卡池(共约 **1.5 万卡**)同时承载两类请求,但二者**本质是同一种工作负载——大模型推理**,差别只在请求来源和模型种类:

| | vibecoding(在线) | 数据生产(离线) |
|---|---|---|
| 形态 | 研发同事用 **Claude Code 调用自托管 GLM5.1** 推理服务 | 批量推理:Caption、QA、评测、清洗、蒸馏等 |
| 模型 | GLM5.1(单一在线模型) | 多模型:GLM5.1、Qwen3.5VL、gpt-oss、MinerU 等 |
| 延迟要求 | 低延迟(TTFT / 生成速度敏感) | 无实时要求,可无限排队 |
| 负载形态 | 突发、平均占用率低,有大量空泡 | 稳定吞吐,专门"填谷" |
| 规模 | 峰值约 **3000 卡**算力,深夜低谷约 **64 卡** | 常驻约 **1.2 万卡** + 在线缩容腾出的卡 |

### 1.2 核心洞察

这套负载有三个决定性特征,直接决定了架构:

1. **vibecoding 是推理服务,不是"给人分卡"。** 容量单位是"在线几个 GLM5.1 副本、能扛多大并发",扩容 = 部署新副本(加载几百 G 权重,冷启动分钟级)。所谓"排队 ≤2 分钟"只在"临时拉副本扛尖峰"时有意义,日常 SLO 实为 TTFT / 生成速度。

2. **GLM5.1 是横跨在线与离线的同一资源。** 数据生产里也有大量 GLM5.1 批量调用。在线请求的空泡(同事读代码、思考时 GPU 闲置)正好可被 GLM5.1 批量请求填满——**二者是正和,不是零和**。

3. **峰谷比极大(3000 : 64 ≈ 47 倍)。** 固定时间窗口"到点切"是错的;在线副本应跟随实时负载**自动扩缩**,腾出的卡自动回流离线池。

### 1.3 重新定位后的问题陈述

> 把 1.5 万卡建成一个**统一的多模型推理算力池**。GLM5.1 单部署、在线与离线在推理层按优先级混跑;在线副本按实时负载在 64–3000 卡间自动扩缩;其余卡是多模型离线池,按需装载不同模型跑幂等分片任务。**唯一硬不变量:GLM5.1 在线副本数始终够扛当前在线 QPS;其余一切服务于离线吞吐最大化。**

## 2. 三层架构

整个系统是三层,而非一个统一的抢占式 HPC 调度器:

```
┌─ 第一层:推理混跑层(吃掉了原需求 ~80% 的复杂度)
│   GLM5.1 单部署。在线(Claude Code)= 高优先级实时插队;
│   GLM5.1 批量生产 = 低优先级填谷。靠推理引擎的连续批处理,
│   同一批副本里在线 SLA 不掉、GPU 利用率压满。不发生搬卡。
│
├─ 第二层:在线副本自动扩缩(真正的关键路径)
│   GLM5.1 在线副本按 TTFT/队列深度在 64 → 3000 卡间 autoscale。
│   缩容腾出的卡自动回流离线;扩容优先回收"正在跑 GLM5.1 批量"
│   的温卡(免换装、秒级)。早高峰前预测性预热,缩容带迟滞防抖。
│
└─ 第三层:多模型离线编排
    其余 ~1.2 万卡按需装载 Qwen3.5VL / gpt-oss / MinerU 等。
    "换装" = 卸载旧模型 + 加载新模型(分钟级,可 drain)。
    离线任务用幂等分片队列,被打断就换机从未完成分片续跑。
```

---

## 3. 核心不变量

以下规则在任何时间、任何负载下都成立,是整套系统的地基:

1. **在线 SLA 优先**:GLM5.1 在线副本数始终不低于"扛住当前在线 QPS"所需的量。这是唯一的硬保障。
2. **混跑不搬卡**:GLM5.1 的在线与离线在推理层靠优先级共存,在线请求实时插队,离线请求填谷;**不靠物理搬卡/抢占**来调和二者。
3. **温卡优先**:在线扩容时,优先回收本就在跑 GLM5.1 批量的卡(权重不用换,秒级转在线),其次才回收需换装的卡。
4. **离线幂等**:离线任务以分片为最小单位、幂等可重入;被中断不需要保存模型状态,换机从未完成分片继续即可。
5. **时间窗口仅作兜底**:autoscale 为主;时间窗口只用于①预测性预热提示 ②人工保障期。

## 4. 资源模型与数字

### 4.1 总量与分配

```
总卡池:≈ 15,000 卡

GLM5.1 在线副本:autoscale 64 → 3000 卡
  下限 64:深夜兜底,扛零星加班
  上限 3000:工作日早高峰峰值
  低谷腾出的卡 → 自动回流离线池

离线池:≈ 12,000 卡(常驻) + 在线缩容腾出的卡
  多模型:GLM5.1(批量)/ Qwen3.5VL / gpt-oss / MinerU / …
  同一张卡同一时刻只装一个模型
```

### 4.2 与原文配额的差异

原文设"工作时间 vibecoding 保底 60%(=9000 卡)",与真实峰值 3000 卡**差一个量级**,属凭空设定,已废弃。本版不再用"业务百分比"做配额,**调度对象是"各模型的在线副本数"**,由 autoscaler 按实时负载决定。

## 5. 典型使用场景

本章把架构落到可验证的具体场景。每个场景标注了它对应原文的哪个场景,以及在新架构下"系统行为"有何不同——关键变化是:**几乎没有"搬卡式抢占",取而代之的是混跑填谷、自动扩缩、温卡回收、换装与幂等分片续跑。**

### 5.1 场景一:工作日早高峰,在线负载爬升(对应原 5.1)

**描述**:周一至周五约 09:00 起,研发同事陆续打开 Claude Code,GLM5.1 在线 QPS 从夜间低位快速爬升至高峰。

**系统行为**:
1. **预测性预热**:autoscaler 按历史曲线在高峰前(如 08:30)提前拉起部分在线副本,避免冷启动落在请求堆积窗口。
2. 实际请求涌入时,推理层先**压低 GLM5.1 批量**(填谷请求让位),腾出在线副本算力。
3. 若仍不足,**温卡回收**:把正在跑 GLM5.1 批量的卡秒级转为在线副本(权重不用换)。
4. 仍不足时才**从离线池回收卡换装**为 GLM5.1 在线副本(分钟级)。

**关键点**:扩容尽量走"压批量 → 温卡 → 换装"的递进,换装是最后手段;预热让大部分扩容不落在关键路径上。

**验收**:TTFT P95 达标;温卡扩容秒级、换装扩容 ≤ 分钟级。

### 5.2 场景二:日内平稳期,在线空泡被离线填满(核心日常,对应原 5.8)

**描述**:白天大部分时间在线 QPS 中等,在线副本存在大量空泡(同事读代码、思考时 GPU 闲置)。

**系统行为**:GLM5.1 批量请求以低优先级**自动填满在线副本的空泡**,GPU 利用率压满,在线 SLA 不受影响。同一批副本同时服务在线与离线,**不发生任何搬卡**。

**关键点**:这是"不搬卡"理念的核心日常形态,也是原文 80% 复杂度被消解的根源。

**验收**:整池利用率 ≥ 85%;在线 SLA 违约率 ≤ 1%。

### 5.3 场景三:突发尖峰,某团队集中使用(对应原 5.3)

**描述**:某团队集中做版本迁移、批量 Agent 编程,在线 QPS 在短时间内翻几倍。

**系统行为**:**不是"抢占数据生产"**。推理层先把 GLM5.1 批量压到最低 → 温卡全部回收转在线 → 仍不够再换装离线卡。被压低或被回收的离线请求**自动回到分片队列,不丢失**。

**关键点**:尖峰是 autoscale 上界(3000 卡)以内的事件;真正的约束是换装速度,因此日常应预留一定温卡缓冲,让尖峰大概率走"压批量 + 温卡"即可吸收。

**验收**:扩容就位时间达标;尖峰期间在线 SLA 不破线。

### 5.4 场景四:工作日夜间,在线缩容、卡回流离线(对应原 5.2)

**描述**:约 19:00 后在线 QPS 下降,直至深夜接近低谷(约 64 卡兜底)。

**系统行为**:
1. autoscaler **带迟滞地**缩减在线副本(不因短暂下探就缩容)。
2. 腾出的卡**自动回流离线池**,装载为 Qwen3.5VL / gpt-oss / MinerU 等跑批量。
3. 深夜缩至 64 卡兜底,仍能响应零星加班请求。

**关键点**:缩容防抖是重点;回流全程自动化,无需人工切窗口。

**验收**:夜间整池利用率 ≥ 90%;在线↔离线转换抖动次数低。

### 5.5 场景五:周末,离线吃满(对应原 5.4)

**描述**:周末在线使用量极低。

**系统行为**:在线维持兜底档(64 卡级,autoscale 仍响应零星请求);其余近 1.5 万卡全部服务离线,调度**按模型聚合换装**以减少切换开销,集中跑大批量 Caption / QA / 评测 / 蒸馏。

**验收**:周末整池利用率 ≥ 90%;离线产出达计划量。

### 5.6 场景六:多模型 + 多卡型离线编排(对应原 5.5)

**描述**:离线侧需同时跑 GLM5.1 批量、Qwen3.5VL caption、MinerU PDF 后处理、gpt-oss 评测,卡型为 B3/B4 混合。

**系统行为**:调度**按模型聚合、按卡型成组**装载副本;对大显存/高带宽强依赖的模型优先分配 B4,可降级任务用 B3;尽量整机成组以减少换装次数与碎片。

**关键点**:这里的"调度"对象是"各模型的副本数与落卡位置",不是业务百分比。

**验收**:换装开销占比低;分卡型利用率与碎片率可观测。

### 5.7 场景七:离线任务被中断后续跑(对应原 5.6,已 reframe)

**描述**:某 Qwen3.5VL caption 批量任务正在运行,早高峰需要回收其占用的卡换装为 GLM5.1 在线副本。

**系统行为**:
1. 对该批离线副本走 **drain**:停止派发新分片,在途分片跑完(或丢弃当前分片)。
2. 卡换装为 GLM5.1 在线副本。
3. 被中断的**未完成分片回到分片队列**,换台机器继续。
4. **无需保存模型状态,无 checkpoint**;只要求"结果写入幂等、分片可重入"。

**关键点**:幂等分片 = 近乎瞬时释放,彻底取代了原文的 checkpoint 状态机。

**验收**:分片完成率 ≥ 95%;重复计算比例可统计且持续降低。

### 5.8 场景八:有截止时间的离线任务(对应原 5.7,已 reframe)

**描述**:某评测数据集必须当天产出,属离线任务但有明确 deadline。

**系统行为**:
1. 在**离线池内部**按紧迫度优先调度该任务的分片,并优先保证其副本不被换装走。
2. **但不侵占在线副本**——在线 SLA 是硬不变量,deadline 任务只在离线池内竞争。
3. 必要时管理员开"人工保障窗口",临时抬高离线常驻量(唯一能突破自动策略的逃生口,需留痕)。

**关键点(需明示的取舍)**:deadline 任务碰不了在线。若白天在线占用极高、离线余量不足,该任务可能需靠夜间窗口或人工保障兜底——**这意味着极端情况下 deadline 与在线 SLA 二选一时,在线优先**。需与业务方提前对齐此红线。

**验收**:离线 deadline 达成率 ≥ 90%;对在线 SLA 零影响。

## 6. 详细需求

### 6.1 推理混跑层

| 编号 | 需求 |
|---|---|
| R1.1 | GLM5.1 采用**单一部署**承载在线与离线两类请求,不拆成两套实例。 |
| R1.2 | 推理网关支持**请求级优先级**:在线(Claude Code)高优先级实时插队,GLM5.1 批量低优先级填谷。 |
| R1.3 | 依赖推理引擎的**连续批处理 / 优先级调度**能力,保证在线 SLA 不被批量请求拖累。 |
| R1.4 | 批量请求在在线高峰可被自动压低甚至暂停,在线低谷自动放量。 |

### 6.2 在线副本自动扩缩(关键路径)

| 编号 | 需求 |
|---|---|
| R2.1 | GLM5.1 在线副本按 **TTFT / 队列深度 / 并发**自动扩缩,区间 64–3000 卡。 |
| R2.2 | **温卡优先回收**:扩容时优先把正在跑 GLM5.1 批量的卡转为在线副本(免换装,秒级)。 |
| R2.3 | **预测性预热**:按历史负载曲线在早高峰前提前拉起副本,避免冷启动落在请求堆积窗口内。 |
| R2.4 | **缩容带迟滞 / 冷却**:避免 QPS 短暂下探就缩容、随后又扩容造成的抖动与无谓换装。 |
| R2.5 | 缩容腾出的卡**自动回流**离线池,无需人工干预。 |

> 注:R2.2–R2.4 是这套架构真正的工程难点。47 倍峰谷比下,决定 vibecoding 体验的不是"抢没抢到卡",而是"副本能不能在 QPS 涨上来之前就位"。

### 6.3 多模型离线编排

| 编号 | 需求 |
|---|---|
| R3.1 | 离线池按需把卡**装载为指定模型副本**(Qwen3.5VL / gpt-oss / MinerU 等)。 |
| R3.2 | "换装" = 卸载旧模型 + 加载新模型,分钟级,**支持 drain**(先停新请求、跑完在途再换)。 |
| R3.3 | 离线任务以**幂等分片队列**组织:读分片 → 推理 → 写结果;被中断换机从未完成分片续跑。 |
| R3.4 | 调度按模型聚合换装(减少换装次数),并尽量**整机/同卡型成组**装载以降低碎片。 |
| R3.5 | 离线任务声明:模型、卡型需求、卡数、预计分片数、(可选)截止时间。**无需声明 checkpoint 能力**。 |

### 6.4 缓存亲和性(Claude Code 专项)

| 编号 | 需求 |
|---|---|
| R4.1 | Claude Code 是 agentic 负载:单任务几十上百次调用、长 context、反复读同一 repo。**KV / prefix cache 命中率直接决定吞吐与延迟**。 |
| R4.2 | 在线请求路由应支持**会话/前缀亲和**,把同一会话或同一 repo 的请求尽量路由到持有相应 prefix cache 的副本。 |
| R4.3 | 原文将"KV Cache 亲和性"划到本期之外(原 14.2),**本场景下应作为一等需求纳入本期**。 |

### 6.5 卡型(B3/B4 等)

| 编号 | 需求 |
|---|---|
| R5.1 | 不同卡型(B3/B4 等)独立统计可用量;任务/模型声明 required / preferred 卡型与可降级策略。 |
| R5.2 | 把大显存/高带宽卡型优先留给对其强依赖的模型,避免被可降级任务占满。 |
| R5.3 | 卡型维度的利用率、排队、换装次数可观测。 |

### 6.6 可观测性(最先建好)

| 编号 | 需求 |
|---|---|
| R6.1 | 在线:TTFT、生成速度、QPS、队列深度、副本数、扩缩容时长、prefix cache 命中率。 |
| R6.2 | 离线:各模型吞吐、分片完成率、换装次数与耗时、单位卡产出。 |
| R6.3 | 资源池:整池利用率、各模型实时占卡曲线、碎片率、卡型分布。 |
| R6.4 | 扩缩容/换装事件全程留痕,支持回放用于调参。 |

## 7. 验收指标

### 7.1 在线(GLM5.1 / Claude Code)

| 指标 | 目标 |
|---|---|
| TTFT P95 | ≤ 目标值(按引擎基线定,建议秒级) |
| 生成速度(tokens/s) | ≥ 目标值 |
| 在线扩容就位时间(温卡回收) | 秒级 |
| 在线扩容就位时间(需换装) | ≤ 分钟级 |
| 在线请求被批量拖累导致的 SLA 违约率 | ≤ 1% |
| prefix cache 命中率 | 可观测并持续优化 |

### 7.2 离线(多模型批量)

| 指标 | 目标 |
|---|---|
| 整池 NPU 平均利用率 | ≥ 85%(含在线空泡被填) |
| 离线分片完成率 | ≥ 95% |
| (有截止时间任务)deadline 达成率 | ≥ 90% |
| 换装开销占比(换装时间 / 总运行时间) | 尽量低,可统计 |
| 单位卡日产出(token / 样本) | 可统计并持续提升 |

### 7.3 资源池

| 指标 | 目标 |
|---|---|
| 整体空闲卡比例 | 尽量低(在线空泡应被离线填满) |
| 在线↔离线卡转换抖动次数 | 可观测,带迟滞后应较低 |
| 碎片率 / 卡型利用率 | 分卡型独立统计 |

## 8. 关键技术前提(需先验证)

本架构的成立依赖以下能力,建议在方案设计前先验证:

1. **推理引擎支持在线/离线优先级混跑**:同一部署内,高优先级在线请求能实时插队,低优先级批量自动填谷且可被压制(vLLM / SGLang 等是否满足、配置如何)。
2. **GLM5.1 批量副本能秒级转在线**:正在跑批量的卡,能否在不卸载权重的前提下快速接管在线请求(R2.2 温卡回收的前提)。
3. **换装(切模型)的真实耗时与 drain 机制**:卸载+加载一个离线模型的实际分钟数,决定 R3.2 与扩容上界。
4. **prefix/KV cache 在多副本下的亲和路由**:R4.2 是否有现成支持。

---

## 9. 需求边界

### 9.1 本期解决

1. GLM5.1 单部署、在线+离线推理层优先级混跑。
2. 在线副本按负载自动扩缩(含温卡回收、预测预热、缩容防抖)。
3. 多模型离线池按需换装 + 幂等分片队列。
4. Claude Code 的 prefix/KV cache 亲和路由。
5. 卡型独立统计与降级。
6. 覆盖在线/离线/资源池三层的可观测性。

### 9.2 本期不解决

1. 跨地域 / 跨云调度。
2. 精细到单用户的商业化计费。
3. 完全自动化的成本优化。
4. 训练型长作业的 checkpoint/恢复(当前无此类负载;若未来引入大规模蒸馏训练再单独立项)。

## 10. 与原始需求的差异说明(供对齐)

原文按"两类业务抢同一池卡"的 HPC 框架设计,以下机制在本场景下**不再需要**,原因如下:

| 原文机制 | 处置 | 原因 |
|---|---|---|
| 保底/弹性/最大三层百分比配额(原 9 章) | 移除 | 配额对象错了。真正要调的是"在线副本数",由 autoscale 决定,不是业务百分比;且原"vibecoding 保底 60%/9000 卡"与真实峰值 3000 卡差一个量级。 |
| checkpoint 状态机(原 5.6 / 6.x) | 移除 | 离线是幂等分片推理,被中断换机续跑没做完的分片即可,无需保存模型态。仅训练型长作业才需要,本期无此负载。 |
| 抢占顺序 P4>P3>P2、被抢占恢复(原 10 章) | 移除 | GLM5.1 在线/离线靠推理层优先级混跑,不发生搬卡式抢占;离线换装本身可 drain、可续跑。 |
| 固定分时窗口硬切(原 8 章) | 降级为兜底 | 47 倍峰谷比下应按负载 autoscale;时间窗口仅用于预测预热与人工保障期。 |
| 反饥饿 / 多次抢占提优(原 15.5) | 移除 | 离线吞吐由"在线空泡 + 常驻 1.2 万卡"自然兜底,不存在被持续抢占饿死的结构。 |

**新增/上提的重点**(原文缺失或低估):
- 在线副本**自动扩缩 + 温卡回收 + 预测预热**——这才是真正的关键路径与工程难点。
- **prefix/KV cache 亲和**——对 Claude Code 这类长 context、反复读同一 repo 的负载是一等需求,原文却划到本期之外。
- **在线/离线推理层优先级混跑**——这是把原文 ~80% 复杂度直接消解掉的核心手段。

---

## 11. 一句话总结

> 这不是一个"两类业务抢卡"的调度系统,而是一个**统一多模型推理算力池**:GLM5.1 单部署让在线(Claude Code)与离线批量在推理层按优先级混跑、在线低谷被离线填满;在线副本按实时负载在 64–3000 卡间自动扩缩、扩容优先回收温卡;其余卡按需装载多模型跑幂等分片离线任务。核心保障只有一条——在线副本始终够扛当前 QPS,其余全部服务于离线吞吐最大化。
