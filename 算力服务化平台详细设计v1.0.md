## 1. 实现范围与替身策略

第一版（贵阳试点）聚焦**离线批处理 + 调度核心 + 治理**这条最能验证混部收益的主线。不依赖昇腾硬件即可全量测试——真实组件用确定性替身：

| 真实组件（设计 §4.2） | 实现替身 | 替身保真点 |
|---|---|---|
| vLLM-Ascend / MindIE | `MockEngine` | 本地 in-process `generate`、加载语义、可触发确定性失败 |
| 统一对象存储 | `LocalObjectStore` | 原子写（tmp→rename）、range read |
| Redis / PostgreSQL | `SqliteShardQueue` | 原子 claim、租约过期、CAS 提交 |
| Volcano / KEDA | `controller` / `preemption` 纯函数 | 伸缩与抢占的**决策逻辑**（不含 K8s 编排） |

> 替身只换"基础设施实现"，不换"机制逻辑"——队列协议、幂等、抢占续跑、计量规则都是真代码、真测试。

## 2. 数据模型（`models.py`）

| 类型 | 字段要点 | 设计依据 |
|---|---|---|
| `QoS` | GUARANTEED / PREEMPTIBLE / BEST_EFFORT_FIXED | §7.1 |
| `CardType` | B3(64G/强) / B4(32G/弱) | §9 异构分池 |
| `JobStatus` | validating→splitting→running→succeeded / completed_with_errors / failed / cancelled | §6.2 含"部分成功"态 |
| `ShardStatus` | pending / running / done / dead / cancelled | §6.3 分片生命周期 |
| `ModelSpec` | name, card_type, cards_per_worker, load_seconds, throughput_rows_per_sec | §4.3 模型注册表 |
| `Shard` | shard_id, job_id, model, file, **start_offset/end_offset/rows**, status, retries, lease_owner, lease_expire_at | §6.2 切指针不切数据 |
| `Job` | job_id, tenant, model, input_key, output_prefix, template_hash, total_shards, client_token, qos | §6.2 |

`Shard` 是指针（file + 字节区间 + 行数），不含数据本体——这是"切 TB 数据只产 MB 元数据"的落点。

## 3. 对象存储（`objectstore.py`）

`LocalObjectStore` 提供存储面契约：

- `write(key, bytes)`：**原子写** —— 写 `key.tmp` → `fsync` → `os.replace`。保证最终路径要么完整要么不存在（§6.3 幂等提交前提）。
- `read_range(key, start, end)`：worker 只取自己那段分片字节（§6.2）。
- `list(prefix)`：跳过 `.tmp`，结果排序确定。
- key 防逃逸根目录。

## 4. 分片队列与租约协议（`queue/`，系统心脏）

### 4.1 接口契约（`base.py`，§4.4 可插拔扩展点）

```
enqueue_many(shards) -> 新增数      # 幂等（按 shard_id 去重）
claim(model, lease) -> Shard|None   # 原子领取；先回收过期；None=没活
renew(shard_id, token, lease) -> bool   # 心跳续租（CAS）
commit(shard_id, token) -> bool         # 提交完成（CAS）
abort(shard_id, token) -> bool          # 放弃（被抢占，不计重试）
fail(shard_id, token) -> bool           # 确定性失败（retry+1，超限→死信）
reap_expired() -> 回收数                 # 过期租约 retry+1→pending / 超限→dead
stats / cancel_job / dead_letters / retry_dead_letters
```

### 4.2 分片状态机

```
          enqueue                 claim
   (无) ───────────► pending ───────────► running ──commit──► done
                       ▲                    │
        abort / 租约过期 │ retry+1            │ fail / 租约过期
                       └────────────────────┤
                                            │ retry > max
                                            ▼
                                          dead ──retry_dead_letters──► pending
   cancel_job: pending|running ─────────► cancelled
```

### 4.3 原子性实现（`sqlite_queue.py`）

- **claim**：`BEGIN IMMEDIATE` 写事务内 —— 先 `_reap_locked`（回收过期）→ `SELECT ... WHERE model=? AND status='pending' ORDER BY retries, shard_id LIMIT 1` → `UPDATE` 置 running + 写入 `lease_owner=uuid` 与 `lease_expire_at=now+lease`。对应 PostgreSQL 的 `FOR UPDATE SKIP LOCKED`。
- **CAS**：renew/commit/abort 都带 `AND lease_owner=? AND status='running'`，`rowcount==1` 才算成功 → 防 A 假死期间被 B 领走、A 复活提交脏状态（§6.3）。
- **时钟可注入**：`self.clock`（默认 `time.time`），测试用可控时钟无需 sleep 即验证租约过期。
- **模型隔离**：claim 按 model 过滤 → 同模型聚批（§7.4）；235B 的分片不会被 small-7B worker 领走。

### 4.4 正确性论证

正确性只依赖"会自动过期的租约"，不依赖 worker 体面退出：被抢占/宕机/OOM/断网，心跳停 → 租约过期 → reap 回收。`kill -9` 与掉电都覆盖。**容错 = 抢占 = 同一码路**（设计 §7 哲学）。

## 5. 切分器（`sharder.py`）

- `rows_per_shard(spec) = target_shard_seconds × throughput_rows_per_sec`（粒度由目标时长反推，§6.2）。
- `split()`：扫描输入字节流的换行偏移 → 按 rows_per_shard 分组 → 每组产出 `Shard(file, byte_start, byte_end, rows)`。
- **确定性**：`shard_id = f"{job_id}-{idx:06d}"`，同输入永远同集合 → 配合 `enqueue_many` 的 `INSERT OR IGNORE` 实现可重入（重复切分零副作用）。
- 守护：超 `max_shards_per_job` 抛错，防误操作打爆队列。

## 6. 推理引擎与 Worker（`inference.py` / `worker.py`）

### 6.1 引擎抽象

`InferenceEngine.{load, generate}`（§4.4 扩展点：vLLM-Ascend / MindIE 通过统一 `generate` 接入）。`MockEngine` 确定性输出 `"[{model}] {row}"`，遇含 `POISON` 的行抛错（驱动死信测试）。

### 6.2 Worker 主循环（§6.3）

```
load() 一次（235B 真实十几分钟，加载成本被后续几百分片摊薄）
loop:
  stop?            → 退出（被抢占：不再领新分片）
  claim → None?    → 退出（饿死：释放卡，§6.4 缩容）
  逐 batch: generate(本地调用) + renew(续租)
  异常             → fail()（毒分片累计重试→死信）
  成功             → 原子写 out/{shard_id}.done → commit(CAS)
```

**幂等三保证**：① 输出 key 由 shard_id 决定、不含 worker 信息 → 重复执行覆盖写；② 先写数据后 commit，commit 永远最后一步；③ commit 带 CAS。被抢占在 batch 边界停、放弃当前分片（租约过期回队列），浪费 ≤ 半个分片。

## 7. 弹性控制器算法（`scheduler/controller.py`，纯函数）

```
desired = min( by_backlog, by_supply, by_quota )
  by_backlog = ceil(work_seconds / target_runtime)   # work_seconds = backlog_rows / throughput
  by_supply  = free_cards // cards_per_worker          # free_cards 已减安全水位
  by_quota   = quota_cards // cards_per_worker
再过加载成本门槛：
  max_by_loadcost = floor(work_seconds / (K × load_seconds))
  desired = min(desired, max_by_loadcost)
  # 防止为一点尾巴活反复起停大实例（235B 加载十几分钟，干五分钟纯亏）
```

`should_add_big_worker(now, predicted_preempt_at, spec, N)`：潮汐末班车——`now + N×load_seconds < predicted_preempt_at` 才放行起大 worker；深夜无预测抢占（None）直接放行。

> 扩缩不对称（§6.4）：扩容靠本函数算目标数提交 worker；缩容靠 worker claim 不到自然饿死，控制器只"不补位"。

## 8. 抢占算法（`scheduler/preemption.py`，纯函数）

`select_victims(workers, cards_needed)`：
1. 只在 `QoS.PREEMPTIBLE` 中选（Guaranteed 在线、BestEffort-Fixed 固定池都不抢）。
2. 排序 `(cards 升序, loaded_seconds_ago 降序)` → **先小后大**（碎卡好回收）、**同规模老的先走**（浪费小、护刚加载完的大实例）。
3. 累加到 `freed ≥ cards_needed` 即停；不够则尽力让路（返回能腾的最大集合）——对应"在线 SLA 优先、批任务无条件让路"。

## 9. 计量结算与配额（`governance/`）

### 9.1 计量（`metering.py`）—— 价格杠杆

```
cost = billable_hours × cards × base_price × card_factor × qos_factor
  card_factor:  B3=1.5, B4=1.0           （异构差异定价）
  qos_factor:   Guaranteed=1.0, Preemptible=0.4   （可抢占 4 折）
  billable:     Preemptible 扣除 preempted_seconds（被抢占时段不计费）
```

把"改造意愿"转成各团队成本账：标 Guaranteed 全价、Preemptible 折价。

### 9.2 配额（`quota.py`）

`reserve / try_reserve / release / available`，卡数硬边界（对应 Volcano 队列 capability）。线程安全。

## 10. 离线作业 API（`batch_api/`）

**服务化边界 = 任务提交**（不是每次推理，§6.1）。五接口（`app.py`）：

| 方法 | 路径 | 行为 |
|---|---|---|
| POST | `/v1/batch/jobs` | 轻校验（模型存在/输入可读/配额够）→ 创建作业 → 切分入队 → 返回 job_id；`client_token` 幂等 |
| GET | `/v1/batch/jobs/{id}` | 从队列 stats 聚合进度，推导 succeeded / completed_with_errors |
| GET | `/v1/batch/jobs/{id}/errors` | 死信分片明细 |
| POST | `/v1/batch/jobs/{id}/cancel` | 取消未完成分片 |
| POST | `/v1/batch/jobs/{id}/retry` | 死信重新入队 |

租户由 `X-Tenant-Id` 标识（缺省 default）。错误码：400 未知模型/输入不可读、429 配额不足、404 作业不存在。`JobStore`（sqlite）持久化作业元数据，`(tenant, client_token)` 唯一索引保证幂等。

## 13. 落地真实环境的替换清单

| 替换点 | 现状 | 真实 |
|---|---|---|
| 推理引擎 | `MockEngine` | vLLM-Ascend / MindIE 适配（实现 `InferenceEngine`） |
| 对象存储 | `LocalObjectStore` | 公司统一对象存储 SDK（同接口） |
| 队列 | `SqliteShardQueue` | Redis(Lua) 或 PostgreSQL（实现 `ShardQueue`） |
| 伸缩执行 | 控制器纯函数 | 输出对接 KEDA ScaledJob + 自研协调器 |
| 抢占执行 | victim 选择纯函数 | 对接 Volcano 抢占 + Pod SIGTERM(宽限 60~120s) |
| worker 编排 | 线程 | K8s Pod / PodGroup（大模型 gang） |
| 跨域 | 单库 | 每地域独立部署 + Karmada（第二版） |

接口已按可插拔设计（§4.4），替换不动机制逻辑。

## 14. 已知简化与后续

- 切分**同步**执行（设计建议亿级异步化）；当前规模够用，后续把 split 移入后台任务队列。
- 弹性控制器是**决策纯函数**，未含 K8s reconcile 编排循环与防抖冷却（留对接 KEDA 时实现）。
- 长度感知切分（按 token 量均衡）、prefix 复用等吞吐优化未实现，属第二版调优。
- 监控指标（利用率/抢占率/SLA/成本/worker 体检）已在计量与队列 stats 留出数据源，未接 Prometheus。
