# 数据工程平台需求文档（v2.0）

> 本版在 v1.0 需求基线之上，纳入五轮新增信息：数据生产规模（40 产线 / 几 PB / 多轮次）、
> 已有平台能力（DataHub / Ray runtime / task 模块 / LiteLLM）、底座选型（ModelArts Lite Cluster）。
> 取代《数据工程平台需求基线v1.0》。配套《[算力服务化平台架构设计v2.0](./算力服务化平台架构设计v2.0.md)》。
> 日期基准：2026-06-14（"1~2 个月见效" ≈ 2026 年 7~8 月）。待确认项标 🔲。

## 1. 项目目标

建设统一算力服务化平台，让**在线推理**与**离线数据生产**共享同一逻辑昇腾算力池，潮汐错峰。四个目标均为必须项：

1. 提高整体算力利用率、降低成本（削峰填谷）
2. 保障在线推理稳定，不受批任务干扰
3. 最大化数据生产吞吐（口径：tokens/天）
4. 多租户配额管理 + 内部成本分摊结算

目标关系：②是硬约束，①③是优化目标，④是化解冲突的经济仲裁机制。

## 2. 业务负载画像

### 2.1 在线推理（Vibe Coding）

- 模型：以 GLM-5.1 为主力（大小模型均有）
- 峰值：白天峰值占全部算力 3 成以内（约 3000 卡量级），夜间低谷
- SLA：有延迟 SLA，可接受轻微波动
- 接入：经统一推理网关（拟用 **LiteLLM**，见 §4）

### 2.2 数据生产（离线批处理）—— 本版重点扩充

- 任务类型：GPU 推理型加工（清洗、打标、合成数据），服务于预训练数据准备
- 时效：无 deadline，纯吞吐导向，是理想弹性填谷负载
- 模型：多模型混用（Qwen3.5-VL-235B、DeepSeek V4 等），大模型加载/切换成本高
- 抢占容忍：部分可抢占；不可抢占部分根因是业务方改造意愿（非技术不可行），占比 🔲 待盘点

## 3. 数据生产的规模与特征（新增，决定架构形态）

| 维度 | 结论 | 架构影响 |
|---|---|---|
| **产线数量** | 约 **40 条产线** | 产线成为一等公民：跨产线公平调度、产线级配额/优先级/成本归集；多产线共享同模型 → 跨产线聚批 |
| **数据量级** | **几个 PB** | 数据局部性从"优化"升级为**硬约束**（PB 不可跨域搬）；存储读带宽成瓶颈；分片元数据爆炸 → 两级切分 + 流式入队 |
| **生产轮次** | 未来数据需**多轮次**生产 | 轮次 = 阶段链（round N 输出 = round N+1 输入）→ 需 **DAG 编排 + 数据血缘/版本**；中间产物≈每轮一份 PB → 需 TTL/GC |
| **吞吐度量** | 已有 tokens/天 统一口径 | 试点验收可直接复用 |

> 多轮 × PB × 40 产线 = 天量 token —— 这是平台存在的根本理由，也让"潮汐填谷 + 可抢占折价"的经济性被放大几个量级。

## 4. 已有平台能力（新增，决定自研边界）

平台不是从零起，已有以下资产，须复用而非重建：

| 能力 | 现状 | 在架构中的角色 |
|---|---|---|
| **DataHub 元数据管理** | 已搭建：数据集、生产元数据、**数据血缘** | 控制面：数据资产**全面治理**（§5）；提供数据集 region 分布（局部性调度用）🔲 待确认是否含物理分布 |
| **Ray runtime** | 已搭建：产线运行 Ray 的执行框架 | 执行面：Ray Data 嵌入式推理 + 多轮阶段执行 |
| **task 管理模块** | 已搭建：简单的生产任务下发 | 接入面：生产**过程管控**唯一入口、自动登记血缘（§5） |
| **LiteLLM** | 拟作网关 + quota | 控制面 + 在线数据面：在线路由、**token/spend 预算**、统一账单 |
| 统一推理网关 | 已有 | 由 LiteLLM 承接/前置 |
| 统一对象存储 | 已有，跨域可访问（受带宽约束） | 存储面：数据/结果/进度 |

**关键澄清——两种 quota 维度**：
- LiteLLM 管 **token / spend / rpm 预算**（在线天然、离线准入校验）。
- **卡级容量配额**（每租户多少卡）由调度器 + Volcano 队列管，LiteLLM 不涉及。
- 统一账：离线卡时折算成 spend 推回 LiteLLM 账本，实现"一个团队一本账"。

## 5. 数据治理与生产过程管控（产线核心需求）

40 产线、几 PB、多轮次的数据生产必须做到**可治理、端到端可追溯、可复现**。两根支柱——**DataHub（数据资产治理）** 与 **task 管理（生产过程管控）**——联动成闭环。

### 5.1 DataHub 全面治理数据

目标：把所有数据集与生产元数据纳入统一治理，作为数据资产的唯一权威账本（system of record）。

- **数据资产目录**：40 产线、多轮次产出的全部数据集统一编目、可检索。
- **数据集版本**：每轮产出独立、不可变版本；round N 输出 = round N+1 输入。
- **数据血缘**：端到端血缘（跨轮次、跨产线），可回溯任一数据集"从哪来、经哪些加工、用哪版逻辑"。
- **生产元数据**：登记产出该数据集的 模型 / prompt 模板版本(hash) / 采样参数 / 产线 / 轮次 / 作业 ID。
- **schema 与质量**：schema 登记；数据质量指标/校验（后续增强）。
- **归属与分级**：owner 团队、敏感分级、用途标签。
- **生命周期治理**：中间产物（每轮≈一份 PB）的保留/归档/TTL。

### 5.2 task 管理 管控生产过程

目标：让 task 管理成为数据生产的**唯一入口与管控点**，所有生产都"过 task"，可观测、可审计、自动登记血缘。

- **唯一入口**：数据生产一律经 task 下发，杜绝旁路产数据（否则血缘断裂、不可治理）。
- **过程管控**：作业/产线的状态、进度、失败、重试、取消全程可见可控。
- **与 DataHub 闭环联动**：
  - 提交时：从 DataHub 读取输入数据集**版本**与血缘，固化到本次作业（保证可复现）。
  - 运行中：记录生产元数据（模型 / 模板版本 / 参数）。
  - 完成时：把输出数据集 + 一条血缘边（输入版本 →[作业:模型+模板+参数]→ 输出版本）**自动写回 DataHub**。
- **多轮编排**：按血缘把 round N→N+1 串成 DAG，前轮产出登记后触发/可触发后轮。
- **可复现**：同一作业（输入版本 + 模板 hash + 参数）重跑产出等价结果。
- **审计**：谁、何时、用什么逻辑、产出了什么，全程留痕。

### 5.3 联动闭环（一句话）

**DataHub 是"数据资产账本"，task 管理是"生产过程关口"**；每次生产 task 从 DataHub 取版本、向 DataHub 回写血缘——二者闭环，实现 40 产线 / PB / 多轮场景下数据的全面治理、端到端可追溯、可复现。

## 6. 基础设施现状

- 总规模：约 10000 卡，昇腾 A2（910B）
- 卡型：**B3**（64G/强）与 **B4**（32G/弱）各约一半；各地域分布 🔲 待盘点
- 地域：**4 个地域**，贵阳约占 1/2（~5000 卡），其余 3 地各约 1/6（各 ~1600 卡）
- 集群：两类业务均在 K8s，当前分开；**现状可重新部署，不构成约束**
- 底座候选：**ModelArts Lite Cluster**（= CCE 托管 K8s + 昇腾 AI 套件：CANN/device plugin/Volcano/MindX/故障自愈），保留 K8s 原生访问 🔲 待确认可用性与"能否挂自研调度插件"
- 跨域网络：专线带宽紧张，大流量须错峰；带宽数值 🔲 待盘点
- 监控：**无统一利用率监控** → 混部前置条件，须先行部署测基线

## 7. 平台用户与运营

- 租户：5~20 个内部团队/业务方 + 40 产线维度；不对外开放
- 结算：内部成本分摊；**已确认接受价格杠杆**——可抢占资源打折计价，固定资源全价
- 安全合规：暂无特殊要求，租户逻辑隔离即可

## 8. 关键约束

1. 第一版 1~2 个月内见效（≈ 2026 年 7~8 月）
2. 技术路线：**拥抱开源 + 华为昇腾栈**，最小化自研
3. 跨地域物理不可合并 → 逻辑统一、物理分治
4. 跨域带宽紧张 + 几 PB → **数据局部性硬约束**：数据在产地加工，跨域大流量错峰/限流
5. 大模型加载成本高 → 调度须 **model-aware**（同模型聚批，跨产线共池）
6. B3/B4 异构 → 分池管理、差异计价（235B 基本只能落 B3）
7. 多轮次 → 需 DAG 编排 + 数据血缘/版本 + 中间产物生命周期管理

## 9. 已闭环决策

| 决策 | 结论 |
|---|---|
| 第一版范围 | **贵阳单机房试点**，验证混部机制与利用率收益 |
| 改造激励 | 价格杠杆：可抢占折价、固定全价 |
| 验收口径 | 批任务吞吐（tokens/天）可量化提升 |
| 技术路线 | 拥抱开源/华为栈（ModelArts Lite、Volcano、MindX、Ray、vLLM-Ascend/MindIE、LiteLLM、DataHub） |
| 离线推理形态 | **嵌入式本地推理**（Ray Data），非服务化 |
| 在线/离线隔离 | 共享卡池 + 控制面，不共享推理引擎实例与数据面 |
| **决策三：冲突裁决** | **保护配额内批任务，仅抢占超配额部分**（非"无条件让路"）→ 三池模型（详见架构 §4.3） |
| **生产唯一入口** | **强制所有数据生产必经 task**（机制约束，杜绝旁路产数据，保证血缘完整） |
| 底座选型 | **ModelArts Lite Cluster**（保留 K8s 原生、可挂自研调度）；待向华为确认插件支持 |
| Ray 部署 | **每地域一个 Ray 集群**（物理分治）；与在线**同 K8s**，priority 抢占可成立 |
| 统一账单 | 离线卡时折算 spend **写入 LiteLLM 账本** |
| 产线/多轮语义 | 产线 = **独立单作业**（无需复杂 per-产线 DAG）；多轮 = **迭代重处理 + 顺序阶段两者都有** → 血缘用统一"数据集版本链" |
