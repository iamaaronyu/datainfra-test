# 算力服务化平台 · 架构设计（v2.0）

> 配套《[数据工程平台需求v2.0](./数据工程平台需求v2.0.md)》。取代《算力服务化平台总体设计v1.0》。
> v2.0 的核心变化：从"自研整套平台"改写为 **"已有件组合 + ModelArts Lite 底座 + 一薄层自研混部"**。
> 本仓库 `src/compute_platform/` 是这薄层策略内核的可测试参考实现（详见《[详细设计v1.0](./算力服务化平台详细设计v1.0.md)》）。
> 待确认/待验证项标 🔲。

## 0. 一页结论

> 平台 70%+ 能力已存在（ModelArts Lite 底座 / Ray runtime / DataHub / task 模块 / LiteLLM）。
> **真正要自研的只有一薄层**：在线/离线**混部仲裁**（QoS + 潮汐抢占 + model-aware）+ **价格杠杆计量** +
> **跨产线公平** + **数据局部性** + **Ray↔抢占胶水**。本仓库的纯函数（controller/preemption/metering）即此层内核。

## 1. 设计原则

1. **共享卡池 + 控制面，不共享推理引擎与数据面**。在线/离线只在物理卡（调度错峰）与治理大脑（配额/计量/血缘）交汇；各跑各的引擎，避免离线大 batch 拖垮在线延迟。
2. **状态外置，算力侧无状态可杀**。状态在队列/对象存储/DataHub，worker 随时可杀 → 容错=抢占=缩容同一码路。
3. **推理不服务化，平台服务化**。离线本地 in-process 推理（拿 batch 控制权冲吞吐）；服务化边界上移到"任务提交"。
4. **买/复用优先，自研只补缺口**。底座与执行交给昇腾栈与已有件，自研集中在混部策略与价格杠杆。

## 4. 各层详解

### 4.1 底座：ModelArts Lite Cluster

CCE（华为托管 K8s）之上预集成昇腾 AI 套件（CANN、NPU device plugin、Volcano、MindX gang/拓扑亲和/故障自愈），且保留 **K8s 原生访问**——这是能在其上跑 KubeRay + 自研调度 + LiteLLM 的前提。
- 🔲 关键验证：**Lite Cluster 是否允许挂自研 Volcano 调度插件 / 自定义 controller**。允许 → 混部层直接落；不允许 → 退回"裸 CCE + 自建昇腾栈"。
- 不选 ModelArts Standard（黑盒全托管，与 Ray + 自研调度 + LiteLLM 三处冲突）。

### 4.2 执行面：Ray + 嵌入式推理

离线执行 = **Ray Data 作业**：
```
ds.map_batches(
    Processor,                       # __init__ load 模型 → 嵌入式推理(加载一次,跨分片摊薄)
    compute=ActorPoolStrategy(...),  # 弹性 worker 池(= 同模型成池)
    num_gpus=8,                      # 235B 占卡;多机用 placement group 做 gang
    batch_size=...)                  # 大 batch 控制权在手 → 离线吞吐高几倍
```
Ray Data 原生提供分片(block)、流式背压、容错(lineage)。**在其上补一层幂等持久进度**：每个输出块落 `{shard_id}.done`（对象存储原子写），使被抢占的活**续跑而非从头重算**（PB + 频繁抢占下 lineage 重算太贵）。这正是本仓库 worker 的幂等提交模式。
- 在线：常驻 vLLM-Ascend/MindIE 实例（W8A8 保延迟），可选 ModelArts 实时服务托管，挂 LiteLLM 后面。

### 4.3 调度面：自研混部仲裁（核心自研）

> **三池模型（已闭环决策三：保护配额内、仅抢超配额）**——这是 v2.0 相对 v1.0 的关键修正。

```
总卡池
├─ 在线保障池        在线峰值预留(~3000)        Guaranteed   全价   不可抢
├─ 离线保障配额池    Σ各租户卡级保障配额          Protected    全价   不可抢(在线也不抢)
└─ 弹性/突发池       其余 = 离线超配额部分        Preemptible  折价   先被抢、也用来填谷
```

约束:**在线保障池 + Σ离线保障配额 ≤ 总卡** → 在线总能扩到峰值,而批任务在配额内有"不被打扰"的保障;**只有超配额的突发部分参与抢占与填谷**。价格杠杆不变:配额内全价、突发折价——想省钱就把活推进突发池。

Volcano/MindX 只给**原语**（优先级、抢占、队列、gang）；**策略**自研。本仓库纯函数即内核：

| 策略 | 内核（src） | 作用 |
|---|---|---|
| QoS 分级 | `models.QoS` | Guaranteed(在线) / Protected(离线配额内,全价不可抢) / Preemptible(离线超配,折价可抢) |
| 弹性伸缩 | `scheduler.controller.desired_workers` | desired=min(积压,空闲卡,卡配额)+加载成本门槛 K → 驱动 KubeRay max replicas |
| 潮汐末班车 | `scheduler.controller.should_add_big_worker` | 预测抢占窗口前 N×加载时间不补大 worker |
| 抢占 victim | `scheduler.preemption.select_victims` | **只抢超配额的 Preemptible worker**、先小后大、护新加载的大实例 |

> 落地要点:worker 按"该租户已用是否超其卡配额"动态打 Protected / Preemptible 标签——配额内的 worker 受保护,超配额的才进抢占候选。`select_victims` 逻辑不变(只挑 Preemptible),变的是打标策略。

**抢占流（Ray↔抢占胶水）**：
```
在线QPS↑ → 在线HPA要卡 → 混部控制器在【弹性/突发池】select_victims(只动超配额 worker)
  → 缩 KubeRay 集群 max replicas + K8s 高优先级抢占该 offline pod(SIGTERM, grace 60~120s)
  → Ray actor 在 batch 边界停 + 落 done-marker → 退出释放卡 → 绑给在线
  → 未完成 block 下次有空闲卡续跑(从 done-marker 之后)
  ※ 离线保障配额池不参与,批任务在配额内不受早高峰影响
```

**model-aware 聚批 + 跨产线共池**：同模型分片排同队列、跨 40 产线共用已加载该模型的 Ray actor 池（产线越多收益越大）。

### 4.4 控制面：DataHub + LiteLLM + 自研治理

- **DataHub（数据资产治理，需求 §5.1）**：全部数据集目录、不可变版本、端到端血缘（跨轮次/跨产线）、生产元数据（模型/模板hash/参数）、归属分级、生命周期；并提供 dataset 的 region 物理分布给局部性调度（✅ 已确认已记录，可直接用）。
- **LiteLLM**：在线路由 + **token/spend 预算** + 统一账本。
- **自研**：① 价格杠杆计量（`metering`：卡时→差异/折价 spend → 推 LiteLLM 账本 + 写 DataHub 记生产成本）；② 卡级容量配额（`quota` + Volcano 队列 capability）；③ 混部监控（利用率/抢占率/SLA违约/成本，先行部署测基线）。

**DataHub ↔ task 治理闭环（需求 §5.3，✅ 已联动）**：task 提交时从 DataHub 取输入数据集版本+血缘并固化；完成时把输出数据集 + 血缘边（输入版本 →[模型+模板+参数]→ 输出版本）自动写回 DataHub。task 是生产唯一入口（✅ 已定为机制约束，强制必经 task），从而保证 40 产线 / PB / 多轮场景下数据可治理、可追溯、可复现。第一版只需在此闭环上**接入调度与计量**，无需重建联动。

### 4.5 接入面：task 模块 + LiteLLM

- 离线：**task 管理模块**是生产**唯一入口与过程管控点**（需求 §5.2，✅ 强制必经 task）——下发作业（声明 数据集版本 + 模型 + 处理逻辑 + 配额）→ 提交 RayJob；提交时读 DataHub 取版本/血缘/region，完成时回写血缘；状态/进度/重试/取消全程可控可审计。
- 在线：LiteLLM 网关（同步/SSE/限流）。
- 服务化边界 = 任务提交，不是每次推理。

### 4.6 存储面

统一对象存储承载数据/结果/done-marker；DataHub 承载元数据/血缘。多轮中间产物按 round 版本管理 + TTL/GC（复用 mvp-file-inference cleanup 思路）。

## 5. 三个规模专题

### 5.1 几 PB（当前总量）→ 数据局部性硬约束
PB 不可跨域搬 → 调度器从 DataHub 读 dataset region 分布（✅ 已记录），**强制把 Ray 作业调到数据所在域**（每地域一个 Ray 集群，✅）；瓶颈含存储读带宽，compute 与 storage 就近；分片元数据用**两级切分**（文件组→block）+ 流式入队，避免万亿分片打爆队列。
> 注:几 PB 是**当前总量**,多轮成倍叠加 → 中间产物 TTL/GC 须认真规划（§4.6）。

### 5.2 40 产线 → 产线一等公民（产线 = 独立单作业）
产线 = **独立单作业**（✅，无需复杂 per-产线 DAG）；作业层级简化为 `产线/作业 → 分片`。跨产线加权公平（权重=配额+积压）；**同模型跨产线聚批**（产线越多收益越大）。

### 5.3 多轮次 → 版本链 + 轻量轮次衔接（治理闭环）
多轮 = **迭代重处理 + 顺序阶段两者都有**（✅），但二者都统一表达为**数据集版本链**：每轮 = 一次作业产出一个新版本，round N 输出版本即 round N+1 输入版本。无需重型 DAG 引擎——**轮次衔接 = 前轮产出登记 DataHub 后触发/可触发后轮**（task 模块承接，✅ 已与 DataHub 联动）。每轮内仍是 Ray Data 嵌入式池；DataHub 记全链路血缘与版本 → 任一数据集可回溯"从哪来、经哪些加工、用哪版逻辑"，且可复现（需求 §5）。

## 6. 自研 vs 复用边界（省工作量总账）

| 模块 | 结论 |
|---|---|
| 昇腾 K8s / 驱动 / CANN / gang / 故障自愈 | ❌ 不自研 → ModelArts Lite |
| 分片 / 批 / 流式 / 容错执行 | ❌ 不自研 → Ray Data |
| 数据集 / 血缘 / 版本 | ❌ 不自研 → DataHub |
| 在线网关 / token 预算 | ❌ 不自研 → LiteLLM |
| 作业下发 | ❌ 不自研 → task 模块 |
| 在线 serving | ❌ 复用 / ModelArts 实时服务 |
| **混部仲裁(QoS/抢占/潮汐/局部性)** | ✅ 自研（本仓库内核） |
| **价格杠杆计量 + 卡级配额 + 跨产线公平** | ✅ 自研 |
| **Ray↔抢占胶水 + done-marker 续跑** | ✅ 自研 |
